## Phase 2: Widget Registration System

### Problem

Adding a new widget type requires editing four different functions across three files:
- `measure()` in `layout.py` (15+ if/elif branches)
- `draw_tree()` in `render.py` (20+ if/elif branches)
- `dispatch_event()` in `app.py` — both mouse click and key event branches (10+ if/elif)
- And optionally `do_layout()` in `layout.py`

### Solution

Create a central registry in a new `widgets.py` file that maps widget type names to handler functions:

```python
_MEASURE = {}
_DRAW = {}
_CLICK = {}
_KEY = {}
_LAYOUT = {}

def register(type_name, *, measure=None, draw=None, on_click=None, on_key=None, layout=None):
    ...
```

Each handler is a free function registered alongside the widget factory. The if/elif chains become dict lookups:

```python
# Before (measure):
if node.type == 'button':
    text = str(node.props.get('text', ''))
    w = ...; h = ...
    return w, h

# After:
handler = _MEASURE.get(node.type)
if handler:
    return handler(node, max_w, max_h)
```

**What stays:** The flexbox container logic in `measure()` and `do_layout()` (the box/scrollbox fallback). Only intrinsic-size widgets move to the registry.

**What moves:** All per-type `if node.type == 'X'` branches move to registered handlers. The dispatch_event mouse and key handlers also register — e.g., `register('checkbox', on_click=click_checkbox, on_key=key_toggle)`.

### Files changed

| File | Change |
|---|---|
| New: `widgets.py` | Registration API and all widget handlers |
| `layout.py` | `measure()` becomes a dict lookup with flexbox fallback; remove if/elif |
| `render.py` | `draw_tree()` becomes a dict lookup; remove if/elif |
| `app.py` | `dispatch_event()` click/key branches use registry; remove if/elif |
| `dom.py` | Import and call `register()` for each widget |

## Phase 3: Widget Improvements

### 3a: Textarea cursor navigation

**Problem:** Textarea only appends at end. No cursor movement, no scroll, no wrapping.

**Solution:**
- Track `cursor_x`, `cursor_y` in element props (persisted via the reconciler's `textarea` value-carried-over logic)
- Arrow keys (UP/DOWN/LEFT/RIGHT) move cursor within the text content
- HOME/END move to start/end of line
- Cursor is rendered at cursor position, not at end of text
- When cursor moves past viewport, adjust `scroll_y` to follow
- No soft wrapping (stretch goal: add wrapping in a future release)

### 3b: ScrollBox clickable scrollbars

**Problem:** Scrollbar thumb is cosmetic — clicking on it doesn't scroll.

**Solution:** In the mouse click handler, check if the click is on the rightmost column of a `scrollbox` node. If so, compute which row was clicked relative to the track, and jump `scroll_y` to that position.

```python
# In click handler, after existing per-type checks:
if target.type == 'scrollbox' and event.x == target.screen_x + target.w - 1:
    track_h = target.h
    click_ratio = (event.y - target.screen_y) / track_h
    max_scroll = max(0, target.content_h - target.h)
    target.scroll_y = int(click_ratio * max_scroll)
    self.request_render()
```

### 3c: VirtualList persistent scroll state

**Problem:** `VirtualListClass` is a functional component, so its `useState` scroll position resets when the parent re-renders with a new `Element(VirtualListClass, kwargs)` (the functional component instance is not matched across renders).

**Solution:** Convert `VirtualList` to a `Component` subclass so the reconciler's key-based matching preserves the instance. The component holds `scroll_y` and `view_h` in its `state` dict.

## Phase 4: Stability & Correctness

### 4a: set_state with change detection

**Problem:** `Component.set_state()` does `self.state.update(state_update)` and always re-renders, even if the new values equal the old ones. The `useState` setter already has a simple equality check, but `Component.set_state` doesn't.

**Solution:** Compare each key in `state_update` against the current value. Only call `request_render()` if at least one value actually changed.

```python
def set_state(self, state_update):
    changed = False
    for k, v in state_update.items():
        if self.state.get(k) != v:
            self.state[k] = v
            changed = True
    if changed and self.app:
        self.app.request_render()
```

This doesn't add deep equality (that's a larger change) but prevents needless re-renders from trivial `set_state({'key': same_val})` calls.

### 4b: Clean exit (context manager)

**Problem:** `App.__del__` is unreliable (circular refs through `app` prop) and has bare `except` clauses.

**Solution:** Add `__enter__`/`__exit__` for context manager support. Document `run()` as being inside a `try/finally cleanup()` rather than relying on `__del__`.

```python
def __enter__(self):
    return self

def __exit__(self, *args):
    self.cleanup()
```

Users can now write `with App(Root).run()` or the context manager is used internally by `run()`.

### 4c: Effect deduplication

**Problem:** `_pending_effects` is a flat list. If a component renders multiple times before effects flush, duplicate `(instance, idx)` entries accumulate.

**Solution:** Use a `set` (or check for duplicates) when appending to `_pending_effects`:

```python
# In useEffect:
key = (instance, idx)
if key not in instance.app._pending_effects_set:
    instance.app._pending_effects_set.add(key)
    instance.app._pending_effects.append((instance, idx))
```

Ancillary: clear the set after flush.
